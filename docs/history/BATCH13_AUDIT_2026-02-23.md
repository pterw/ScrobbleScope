# BATCH13 Pre-Approval Audit (2026-02-23)

## Verdict: APPROVED WITH CORRECTIONS

found -- two minor boundary inaccuracies in line references, one incorrect

have been applied directly to `BATCH13_PROPOSAL.md`.

## WP-3 findings (retry extraction)

### `lastfm.py` return protocol

**Claim verified.** `fetch_once()` (L98-130) returns 2-tuple `(data,
retry_after)`. Success condition is `data is not None` (L140). `ValueError`
re-raised bare at L145-146. All match the proposal.

### `spotify.py` return protocol

**Claim verified.** `search_once()` (L51-71) returns 3-tuple `(result,
retry_after, done)` with explicit `done` flag. Batch `fetch_once()` (L111-136)
returns the same 3-tuple pattern.

### Jitter formula

**Claim verified for search.** L84: `(abs(hash((artist, album, attempt))) %
200) / 1000.0`. Exact match.

**Discrepancy found for batch.** The proposal states "no jitter" for the batch
caller, but `fetch_spotify_album_details_batch` at L149 applies jitter:
`(abs(hash((tuple(album_ids), attempt))) % 200) / 1000.0`. The hash input
differs from the search function (uses `tuple(album_ids)` instead of
`(artist, album)`).

**Correction applied:** Updated the proposal's WP-5 batch caller description to
include jitter with the correct hash input.

### Backoff values

**Claim verified for lastfm.** L152: `min(0.25 * (attempt + 1), 1.0)`. Exact
match.

**Claim verified for spotify search.** L90: `await asyncio.sleep(1)`. Fixed 1s
backoff. Matches proposal.

**Discrepancy found for spotify batch.** The proposal states the batch uses
"a fixed `1.0`" backoff, but L155 uses `await asyncio.sleep(2**attempt)` --
exponential backoff (1, 2, 4, ...), not fixed.

**Correction applied:** Updated the proposal's WP-5 batch caller description to
use `backoff_fn=lambda attempt: 2**attempt`.

### Import path

**Claim verified.** Both `lastfm.py` (L13) and `spotify.py` (L14) already
import from `scrobblescope.utils`. Adding `retry_with_semaphore` introduces no
new dependency for either caller.
